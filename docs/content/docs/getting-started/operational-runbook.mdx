## Startup modes

<SignalGrid>
  <SignalCard eyebrow="Demo Mode" title="`make demo`">
    Starts the coordinator with embedded service nodes and the expected demo topology. Use this for realistic local runs that exercise discovery and remote invocation.
  </SignalCard>
  <SignalCard eyebrow="Clean Demo" title="`make demo-clean`">
    Same as demo mode, but deletes the existing SQLite database first. Use this when you want a fresh state for a walkthrough or reproducible local debugging.
  </SignalCard>
  <SignalCard eyebrow="Dev Server" title="`make run`">
    Starts the FastAPI app with reload. Useful when working on the backend itself, but you are responsible for making sure the rest of the environment matches what you want to test.
  </SignalCard>
</SignalGrid>

## Core commands

```bash
cd backend
make install
make demo
make demo-clean
make run
make test
make lint
```

## Ports, files, and paths that matter

| Item | Default |
| --- | --- |
| API port | `8080` |
| Coordinator libp2p listen port | `9100` |
| Embedded service ports | `9200`, `9201`, `9202` |
| SQLite database | `backend/data/quantum_coordinator.db` |
| Example config | `backend/config/config.example.yaml` |

## The fastest health check sequence

<Steps>
  <Step>
    ### Confirm the API is alive

    ```bash
    curl http://127.0.0.1:8080/api/v1/health
    ```
  </Step>

  <Step>
    ### Confirm services are being tracked

    ```bash
    curl http://127.0.0.1:8080/api/v1/services
    ```
  </Step>

  <Step>
    ### Submit or recheck a job

    ```bash
    curl http://127.0.0.1:8080/api/v1/jobs/<job_id>
    ```
  </Step>

  <Step>
    ### Inspect the compiled plan if a `plan_id` exists

    ```bash
    curl http://127.0.0.1:8080/api/v1/plans/<plan_id>
    ```
  </Step>
</Steps>

## What each checkpoint tells you

<DefinitionGrid>
  <Definition term="Health">
    Whether the FastAPI process is responding at all. This does not by itself prove discovery or runtime execution are healthy.
  </Definition>
  <Definition term="Services">
    Whether the registry has live advertisements. In demo mode, an empty list usually means the libp2p or advertisement path is not functioning correctly.
  </Definition>
  <Definition term="Job">
    Whether the application layer accepted the circuit, persisted the record, and is making lifecycle progress.
  </Definition>
  <Definition term="Plan">
    Whether the planner succeeded and whether the current process still has the compiled plan cached for inspection.
  </Definition>
</DefinitionGrid>

## State and recovery expectations

The backend persists:

- jobs
- service advertisement snapshots
- reservations
- fragment execution events

On startup it also:

- runs SQLite migrations
- reloads the registry snapshot
- marks cached services unavailable until they re-advertise
- attempts to recover unfinished jobs

That means a restart preserves important operational data, but not every in-memory convenience artifact.

## Things that are intentionally not durable yet

- the compiled plan cache behind `GET /api/v1/plans/{plan_id}`
- any frontend browser session state outside the browser itself
- long-term time-series fidelity history

## Safe reset patterns

Use `make demo-clean` when:

- you want to erase prior jobs
- you suspect stale local state is confusing a demo
- you want a clean baseline before capturing screenshots or recordings

Use a normal restart when:

- you want to verify unfinished-job recovery
- you want to preserve old job records
- you only need to cycle the API process

## Recommended inspection order when something looks wrong

1. Check `health`.
2. Check `services`.
3. Check whether the frontend is pointed at the right API base URL.
4. Submit a minimal circuit.
5. Inspect the job status and error string.
6. Inspect the plan if one exists.
7. Reset cleanly only after you understand which layer is actually failing.
