<Spotlight eyebrow="Outcome" title="What success looks like">
By the end of this guide you will have the backend coordinator running, embedded service nodes advertising capabilities, the frontend dashboard connected, a submitted job with a `job_id`, a compiled `plan_id`, and a finished result that includes both fragment metadata and Qiskit-backed quantum analysis.
</Spotlight>

## Prerequisites and assumptions

- Python `3.10+`
- [`uv`](https://github.com/astral-sh/uv)
- Node.js `20+` and `npm`
- a local environment where ports `8080`, `9100`, `9200`, `9201`, and `9202` can be opened

<Callout type="warning" title="What the demo script does">
`backend/scripts/demo-start.sh` force-kills anything listening on ports `8080`, `9100`, `9200`, `9201`, and `9202`. The `demo-clean` mode also deletes `backend/data/quantum_coordinator.db`.
</Callout>

<SignalGrid>
  <SignalCard eyebrow="Backend" title="Coordinator and embedded nodes">
    The demo mode starts one FastAPI coordinator plus three embedded service nodes under the same overall project. That is the easiest way to exercise discovery, planning, and execution without provisioning separate hosts.
  </SignalCard>
  <SignalCard eyebrow="Frontend" title="Dashboard workspace">
    The React dashboard is optional but strongly recommended. It exposes job progression, fragment DAGs, node fidelity summaries, and the quantum analysis payload in a much easier format than raw JSON.
  </SignalCard>
  <SignalCard eyebrow="Local-Only Alternative" title="When to disable libp2p">
    If you only need the API flow for a fast development loop, you can run with `QC_LIBP2P__ENABLED=false`. That uses local in-process execution and skips the real network path.
  </SignalCard>
</SignalGrid>

## Step-by-step setup

<Steps>
  <Step>
    ### Install backend dependencies

    ```bash
    cd backend
    make install
    ```

    The first install may take a while because the backend pulls in both Qiskit and the `py-libp2p` dependency chain.
  </Step>

  <Step>
    ### Start the coordinator demo

    ```bash
    cd backend
    make demo
    ```

    For a clean database and a more reproducible local run:

    ```bash
    cd backend
    make demo-clean
    ```

    Under the hood this launches:

    - FastAPI on `http://127.0.0.1:8080`
    - the coordinator libp2p listener on port `9100`
    - embedded service nodes on ports `9200` through `9202`
  </Step>

  <Step>
    ### Start the frontend dashboard

    In a second terminal:

    ```bash
    cd frontend
    npm install
    npm run dev
    ```

    If your API base URL is different, point the frontend explicitly:

    ```bash
    cd frontend
    VITE_API_BASE_URL=http://127.0.0.1:8080 npm run dev
    ```
  </Step>

  <Step>
    ### Verify the coordinator and topology

    ```bash
    curl http://127.0.0.1:8080/api/v1/health
    curl http://127.0.0.1:8080/api/v1/services
    ```

    You should see:

    - a health payload with service name, environment, version, and uptime
    - at least a few service advertisements with node IDs, service types, and fidelity values
  </Step>

  <Step>
    ### Submit a sample distributed circuit

    ```bash
    curl -X POST http://127.0.0.1:8080/api/v1/circuits/submit \
      -H 'Content-Type: application/json' \
      --data-binary '{
        "circuit": "OPENQASM 3;\nqubit[2] q;\nbit[1] c;\nbell_pair q[0], q[1];\ncnot q[0], q[1];\ncz q[0], q[1];\nteleport q[0], q[1];\nsyndrome_extraction q[0];\ndistillation q[1];\nmeasure q[0] -> c[0];"
      }'
    ```

    The response returns a `job_id` immediately with status `QUEUED`.
  </Step>

  <Step>
    ### Poll the job and inspect the plan

    ```bash
    curl http://127.0.0.1:8080/api/v1/jobs/<job_id>
    curl http://127.0.0.1:8080/api/v1/plans/<plan_id>
    ```

    Once the job is executing or completed, you should also see:

    - `plan_id`
    - `progress`
    - partial or complete `fragment_results`
    - a final `quantum_result` after completion
  </Step>
</Steps>

## URLs worth bookmarking during development

- OpenAPI docs: `http://127.0.0.1:8080/docs`
- ReDoc: `http://127.0.0.1:8080/redoc`
- Health: `http://127.0.0.1:8080/api/v1/health`
- Services: `http://127.0.0.1:8080/api/v1/services`

## What a successful first run should reveal

<DefinitionGrid>
  <Definition term="Health endpoint">
    Confirms the coordinator is alive and exposes version, environment, and uptime. If this fails, fix startup before debugging discovery or planning.
  </Definition>
  <Definition term="Services endpoint">
    Tells you whether embedded nodes are actually advertising capabilities. If this is empty in demo mode, the discovery or libp2p path is not healthy yet.
  </Definition>
  <Definition term="Job response">
    Shows the lifecycle moving from `QUEUED` through `COMPILING`, `RESERVING`, `EXECUTING`, and finally `COMPLETED` or `FAILED`.
  </Definition>
  <Definition term="Plan response">
    Proves the planner produced fragments, dependencies, and candidate assignments rather than handing the circuit to a single opaque executor.
  </Definition>
</DefinitionGrid>

## A note on the sample circuit

The example circuit is deliberately mixed. It includes:

- entangling behavior (`bell_pair`, `cnot`, `cz`)
- a logical transfer step (`teleport`)
- logical orchestration steps (`syndrome_extraction`, `distillation`)
- a measurement step (`measure`)

That makes it useful for seeing both the planner's routing behavior and the Qiskit result builder's interpretation limits.

## Running without libp2p

If you only want the API flow without remote transport, disable libp2p and use local in-process gate execution:

```bash
cd backend
QC_LIBP2P__ENABLED=false make run
```

Use this mode when:

- you are iterating on API contracts
- you are debugging docs or frontend wiring
- your local environment cannot support the libp2p dependency stack

Avoid using it when you want to evaluate the distributed coordination story itself.
