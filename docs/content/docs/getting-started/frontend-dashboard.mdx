<Spotlight eyebrow="Purpose" title="A control room for the coordinator">
The frontend is not a marketing shell. It is an inspection workspace for the same backend contracts exposed by the API. Its job is to make topology, job state, plan structure, and quantum-result analysis easier to understand while a workflow is running.
</Spotlight>

## Run it locally

```bash
cd frontend
npm install
npm run dev
```

The app uses `VITE_API_BASE_URL` when set. If you do not provide it:

- it prefers the current origin when the page itself is served from port `8080`
- otherwise it falls back to `http://127.0.0.1:8080`

## Main surfaces in the UI

<SignalGrid>
  <SignalCard eyebrow="Overview" title="Control-plane health and current state">
    The top of the workspace surfaces uptime, service counts, average fidelity, and the current lifecycle phase of the tracked job. It is the quickest answer to "is the system alive and what is it doing right now?"
  </SignalCard>
  <SignalCard eyebrow="Topology" title="Live Service Mesh">
    Service advertisements are grouped by node so you can see capability spread, current availability, listen addresses, and fidelity summaries without manually joining multiple API responses.
  </SignalCard>
  <SignalCard eyebrow="Input" title="Circuit Composer">
    You can edit raw circuit text, load built-in sample circuits, and immediately see basic circuit insight such as detected format, operation count, and service breakdown.
  </SignalCard>
  <SignalCard eyebrow="Planning" title="Execution Graph">
    The plan viewer renders fragments as a DAG with dependencies, primary assignments, fallback nodes, and runtime status overlay. It is the easiest way to understand how the planner routed a job.
  </SignalCard>
  <SignalCard eyebrow="Runtime" title="Fragment drill-down">
    Selecting a fragment reveals candidate scores, routing decisions, chosen node, fallback sequence, and any observed fidelity from execution.
  </SignalCard>
  <SignalCard eyebrow="Analysis" title="Quantum result surfaces">
    The final analysis area exposes counts, measured probabilities, Bloch vectors, entanglement entropy, density matrices, statevector entries, and metadata without forcing users to parse large JSON objects.
  </SignalCard>
</SignalGrid>

## How data stays current

The UI combines several update paths:

- active jobs stream state changes over WebSocket
- job details are polled while a workflow is in flight
- health, services, and node fidelity summaries refresh in the background

That means you can watch a job move from `QUEUED` through `COMPLETED` while the plan and analysis panels fill in.

## Workflow the dashboard is optimized for

<Timeline>
  <TimelineItem eyebrow="Step 1" title="Compose or paste a circuit">
    Start in the circuit editor. The dashboard immediately analyzes the text and extracts a lightweight view of format, qubit count, and service mix before the backend sees anything.
  </TimelineItem>
  <TimelineItem eyebrow="Step 2" title="Submit and track one job">
    The workspace records the submitted `job_id`, streams lifecycle changes, and keeps polling the job until a terminal state is reached.
  </TimelineItem>
  <TimelineItem eyebrow="Step 3" title="Inspect the compiled plan">
    When the backend returns a `plan_id`, the UI loads the plan, draws the fragment graph, and lets you pivot between graph-wide and fragment-specific perspectives.
  </TimelineItem>
  <TimelineItem eyebrow="Step 4" title="Analyze the final result">
    Once the workflow completes, the result panes become a structured lab notebook: measurements, geometry, deep state inspection, and metadata remain available in the same session.
  </TimelineItem>
</Timeline>

## State that survives a refresh

The app persists its working session in browser storage, including:

- the current circuit text
- the tracked job and plan
- selected node and fragment
- pagination state for analysis views

That makes it practical to refresh the page without losing the current investigation.

## When to prefer the dashboard over raw API calls

- You want to see the fragment DAG rather than infer it from JSON.
- You want side-by-side visibility into registry health, plan quality, and final results.
- You are presenting the project to others and need a clearer narrative than `curl` output.

## Things the dashboard deliberately does not hide

<DefinitionGrid>
  <Definition term="Backend truth still wins">
    The dashboard is a visualization layer. It does not invent extra orchestration state or analysis. If something looks odd in the UI, verify it against the raw API rather than assuming the UI is the source of truth.
  </Definition>
  <Definition term="Current-fidelity view">
    Node fidelity summaries are derived from the current registry snapshot, not from a long-lived historical observability pipeline.
  </Definition>
  <Definition term="Plan durability caveat">
    The frontend can only inspect plans that the running backend process still has in memory. If the backend restarts, old `plan_id` lookups can stop resolving even though the job record still exists.
  </Definition>
</DefinitionGrid>

For exact request and response contracts, keep [API Reference](../reference/api-reference.mdx) open beside the dashboard.
