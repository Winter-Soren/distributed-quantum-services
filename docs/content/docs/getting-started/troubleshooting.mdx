<Spotlight eyebrow="Goal" title="Debug the right layer first">
Most local problems come from one of four layers: process startup, service discovery, planner feasibility, or docs/frontend environment mismatch. This page is organized by symptom so you can narrow the fault before resetting everything.
</Spotlight>

## Symptom: the API never starts cleanly

Likely causes:

- a required port is already in use
- the Python environment is incomplete
- `py-libp2p` or another runtime dependency failed during install

What to do:

- use the demo script so it frees the expected ports first
- rerun `make install`
- try `QC_LIBP2P__ENABLED=false make run` to isolate network-related startup problems

## Symptom: `/api/v1/services` is empty in demo mode

Likely causes:

- the embedded service nodes never came up
- advertisements are not being published or ingested
- the libp2p topology is not healthy yet

What to do:

- confirm you started with `make demo` or `make demo-clean`
- wait a few seconds for staggered advertisement publication
- restart in clean demo mode if the topology looks stale

## Symptom: job submission works but planning fails

Likely causes:

- the circuit uses unsupported syntax
- no currently advertised node satisfies a required service type or minimum fidelity
- the registry is present but availability is false for the needed services

What to do:

- try a known-good sample circuit first
- inspect `GET /api/v1/services`
- compare the circuit operations against [Circuit Language](../reference/circuit-language.mdx)

## Symptom: the job stays in progress or eventually fails during execution

Likely causes:

- reservation rejection
- timeout exhaustion
- a simulated node drop
- quality degradation below the minimum fidelity threshold

What to do:

- inspect the job payload for `progress`
- look at `fragment_results` for successful fragments
- use the frontend dashboard to inspect fragment-level status and candidate routing

## Symptom: `GET /api/v1/plans/{plan_id}` returns `404`

This is one of the easiest behaviors to misread.

What it usually means:

- the job record exists, but the backend process that compiled the plan has restarted
- the plan cache is in memory and is therefore not durable across process restarts

What to do:

- treat the job record as durable and the plan endpoint as session-scoped
- rerun the workflow if you need a fresh inspectable plan

## Symptom: the frontend loads but cannot talk to the backend

Likely causes:

- `VITE_API_BASE_URL` points at the wrong API
- the backend is down
- the frontend is open against a different port or host than expected

What to do:

- set `VITE_API_BASE_URL=http://127.0.0.1:8080`
- confirm `/api/v1/health` from a terminal
- reload after restarting the backend

## Symptom: auth or rate-limit errors show up unexpectedly

Likely causes:

- config overrides enabled auth or rate limiting
- the frontend or `curl` request is missing the `X-API-Key` header

What to do:

- inspect your environment overrides
- check `QC_API__ENABLE_AUTH`, `QC_API__API_KEY`, and `QC_API__ENABLE_RATE_LIMIT`

## Symptom: the docs site itself fails to render or build

Docs-specific gotchas:

- unsupported Shiki language tags in fenced code blocks can break rendering
- remote font fetching can break offline builds if fonts are not local
- stale Next build locks can block new production builds

What to do:

- prefer supported fence languages like `bash`, `json`, `python`, or `txt`
- run `npm run types:check` in `docs/`
- rerun `npx next build --webpack` when you want a production verification pass

## Symptom: local state is confusing the demo

Use the clean reset path:

```bash
cd backend
make demo-clean
```

Use this when:

- old jobs are cluttering the database
- you want a clean walkthrough
- you suspect previous runs left the local environment in a confusing state

## When not to reset immediately

Avoid reflexively deleting the database when:

- you are trying to verify restart recovery
- you want to preserve failure evidence
- the problem is clearly in docs or frontend configuration rather than backend state
