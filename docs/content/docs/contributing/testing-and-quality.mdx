<Spotlight eyebrow="Quality Strategy" title="Test the boundary that owns the behavior">
The backend already has a meaningful split between unit and integration tests. Good additions usually follow that structure: keep pure logic cheap to test in isolation, and reserve heavier tests for the boundaries where transport, persistence, or orchestration actually matter.
</Spotlight>

## Main test commands

```bash
cd backend
make test
make lint
```

The lint target currently runs:

- `ruff check`
- `mypy`

## Test suite structure

### Unit tests

Representative areas already covered include:

- configuration loading
- health endpoint behavior
- planner properties
- parser normalization
- dependency DAG logic
- reservation protocol behavior
- registry and registry-store behavior
- SQLite job and runtime storage
- Qiskit result generation
- libp2p fabric units

### Integration tests

Representative areas already covered include:

- job API end-to-end flow
- runtime executor failure handling
- service discovery integration behavior

## Where to add a new test

<DefinitionGrid>
  <Definition term="Pure transformation or validation">
    Put it in unit tests. Parser behavior, planner scoring, state transitions, schema validation, and store serialization are all good unit-test candidates.
  </Definition>
  <Definition term="Cross-module orchestration">
    Use integration tests when the behavior spans planning, runtime, persistence, and API layers together.
  </Definition>
  <Definition term="Network-sensitive behavior">
    Prefer the existing support helpers around libp2p rather than inventing a new ad hoc harness.
  </Definition>
  <Definition term="UI behavior">
    The frontend currently has less formal test infrastructure than the backend, so be more deliberate and at least validate builds plus behavior manually when making UI changes.
  </Definition>
</DefinitionGrid>

## Good backend testing habits in this repo

<Checklist>
  <CheckItem>
    Assert the observable contract, not only internal implementation details.
  </CheckItem>
  <CheckItem>
    Keep tests deterministic by controlling inputs, seeds, and expected topology shape.
  </CheckItem>
  <CheckItem>
    When possible, validate both the final outcome and the intermediate persisted state.
  </CheckItem>
  <CheckItem>
    Prefer extending existing test modules for related behavior so the suite remains discoverable.
  </CheckItem>
</Checklist>

## Gaps that are still worth improving

- broader frontend test coverage
- benchmark and performance regression coverage
- experiment-harness validation once that layer exists
- stronger evidence around the unfinished roadmap items

## A lightweight contribution rule of thumb

- parser, registry, planner, and store changes should almost always come with backend tests
- API route changes should usually come with response-shape or integration coverage
- docs-only changes still deserve `docs` type/build checks
