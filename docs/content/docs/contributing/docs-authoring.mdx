This site is meant to be a primary entry point for users, researchers, and contributors. Good docs changes improve both truthfulness and usability: they clarify real behavior, connect related concepts, and make dense material easier to navigate visually.
</Spotlight>

## Content structure

Docs content lives under:

```text
docs/content/docs/
```

Navigation is driven by nearby `meta.json` files. Page metadata comes from frontmatter.

## Useful frontmatter fields

The current docs schema supports:

- `title`
- `description`
- `icon`
- `full`

Use `icon` when it genuinely helps the sidebar and information scent, not just because it is available.

## Custom MDX components available in this site

In addition to the built-in Fumadocs components, this site maps custom components such as:

- `Spotlight`
- `SignalGrid`
- `SignalCard`
- `Timeline`
- `TimelineItem`
- `DefinitionGrid`
- `Definition`
- `LayerGrid`
- `LayerCard`
- `Checklist`
- `CheckItem`

Use them when:

- the page needs scannable structure
- you want to avoid giant uniform bullet sections
- you want readers to distinguish between definitions, workflows, and layered architecture

## Styling guidance for this repo

- Favor rich but calm presentation over novelty for novelty's sake.
- Keep the docs consistent with the existing ocean-toned visual language.
- Use cards, spotlight panels, and definition grids to break up dense technical material.
- Do not sacrifice readability for decoration.

## Docs-specific gotchas

### Code fences

Not every syntax label is bundled by the current highlighter setup. If a niche language fence causes rendering issues, prefer a supported highlighter label or fall back to `txt`.

### Fonts

This docs app uses local bundled fonts instead of runtime Google font fetching so builds do not depend on external network access.

### AI search

The floating AI search panel only appears when `OPENROUTER_API_KEY` is configured. The docs should still feel complete and navigable without it.

## Commands to run before shipping docs changes

```bash
cd docs
npm run types:check
npx next build --webpack
```

## Writing standards for this handbook

<Checklist>
  <CheckItem>
    Prefer implementation-backed claims over vague summaries.
  </CheckItem>
  <CheckItem>
    Name limitations directly when a capability is partial or still evolving.
  </CheckItem>
  <CheckItem>
    Link related pages so readers can move from overview to detail naturally.
  </CheckItem>
  <CheckItem>
    Balance depth with structure. Dense pages should still be skimmable.
  </CheckItem>
</Checklist>
