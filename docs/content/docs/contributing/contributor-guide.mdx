<Spotlight eyebrow="Contributor Mindset" title="Change one layer with context, not all layers by accident">
This repository has three active surfaces: the backend coordinator, the frontend dashboard, and this docs site. The fastest way to make high-quality changes is to decide which surface you are truly modifying, find the files that own that behavior, and then run the checks that actually validate that surface.
</Spotlight>

## Repository layout

| Path | What lives there |
| --- | --- |
| `backend/src/quantum_coordinator/` | Coordinator source code |
| `backend/tests/` | Unit and integration tests |
| `backend/docs/` | Design, requirements, and implementation-planning documents |
| `frontend/src/` | React dashboard |
| `docs/` | Fumadocs site |

## Mental model before you touch code

<SignalGrid>
  <SignalCard eyebrow="Backend" title="This is the source of truth for behavior">
    If a route, runtime event, result field, or planner decision matters, confirm it in `backend/src/` rather than assuming older Markdown or frontend types are authoritative.
  </SignalCard>
  <SignalCard eyebrow="Frontend" title="This is an interpretation layer">
    The dashboard is important, but it reflects backend contracts. UI improvements should stay aligned with the actual API and runtime model rather than inventing new states casually.
  </SignalCard>
  <SignalCard eyebrow="Docs" title="These should track reality">
    The docs should be generous and detailed, but never aspirational without calling out the limitation. Honest documentation is more valuable than polished fiction.
  </SignalCard>
</SignalGrid>

## Backend workflow

Install and run:

```bash
cd backend
make install
make run
```

Common quality commands:

```bash
cd backend
make lint
make test
make format
```

Useful entry points when reading backend code:

- `api/app.py`: app assembly and routes
- `application/job_manager.py`: job lifecycle orchestration
- `planning/`: parser, DAG building, fragment creation, and scoring
- `runtime/executor.py`: retries, timeouts, and fallback
- `service_discovery/registry.py`: topology freshness and queries
- `infra/persistence/`: durable job, registry, and runtime storage
- `runtime/qiskit_results.py`: final quantum result reconstruction

## Frontend workflow

```bash
cd frontend
npm install
npm run dev
npm run lint
npm run build
```

The frontend is centered around:

- `src/App.tsx` for the main workspace
- `src/lib/quantum-api.ts` for API contracts
- `src/lib/quantum-dashboard.ts` for visualization helpers and formatting logic
- `src/components/` for reusable presentation pieces such as the Bloch sphere and UI primitives

## Docs workflow

```bash
cd docs
npm install
npm run dev
npm run types:check
npm run build
```

Docs content lives in `docs/content/docs/`.

Navigation is driven by:

- `meta.json` files for section ordering
- frontmatter for page titles and descriptions

Site-level behavior lives in:

- `docs/app/` for routes and layouts
- `docs/lib/layout.shared.tsx` for navigation and repo links
- `docs/components/` for shared UI helpers
- `docs/components/docs-kit.tsx` for the richer MDX presentation blocks used across this handbook

## Keeping the docs trustworthy

When you update behavior or interfaces:

- verify claims against `backend/src/` or `frontend/src/`, not only older Markdown files
- keep `backend/docs/` and `docs/content/docs/` aligned when architecture or roadmap claims change
- document current limitations instead of smoothing them over

## A good change strategy

1. Reproduce the current behavior before editing.
2. Identify the owning module or page.
3. Change the smallest set of files that truly own the behavior.
4. Run the checks for the touched surface.
5. Update docs if the behavior or workflow changed.

## Optional AI search setup

The docs site can expose an AI chat panel, but it is optional:

```bash
cd docs
OPENROUTER_API_KEY=your-key npm run dev
```

Without that environment variable, the normal docs experience still works and the AI entry point stays hidden.

## A good contribution checklist

- Change the smallest surface that solves the problem.
- Run the checks for the surface you touched.
- Update docs when behavior, routes, or workflows change.
- Prefer implementation-backed wording over aspirational wording.
