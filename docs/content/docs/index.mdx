<Spotlight eyebrow="System Thesis" title="Treat quantum operations as network-visible services">
Distributed Quantum Services is a research-oriented coordinator for routing, executing, and analyzing quantum-service workflows across a libp2p-connected network. Instead of assuming one monolithic backend owns the entire circuit, the coordinator accepts OpenQASM-like input, turns it into dependency-aware fragments, assigns those fragments to available services, executes them with retries and fallbacks, persists the operational trail, and reconstructs an analysis payload with Qiskit.

The result is a codebase that sits in the space between distributed systems research, orchestration middleware, and quantum workflow experimentation.
</Spotlight>

  <SignalCard eyebrow="For Users" title="Run a distributed workflow end to end">

## What the system actually implements today

<LayerGrid>
  <LayerCard eyebrow="Control Plane" title="Planning and orchestration">
    FastAPI receives circuits, `JobManager` owns the lifecycle, the planner normalizes and fragments workflows, and the reservation layer gates access to services before runtime execution begins.
  </LayerCard>
  <LayerCard eyebrow="Execution Plane" title="Real network coordination">
    Service advertisements travel over pubsub and fragment invocations travel over libp2p request streams. The runtime handles timeout, retry, fallback, and fragment-level event recording.
  </LayerCard>
  <LayerCard eyebrow="Data Plane" title="Inspectable results and recovery">
    SQLite stores jobs, registry snapshots, reservations, and runtime events. Qiskit reconstructs quantum outputs that the frontend and API can expose in a stable, inspectable form.
  </LayerCard>
</LayerGrid>

## What makes this more than a toy demo

<Checklist>
  <CheckItem>
    Jobs are persisted immediately instead of living only in request memory.
  </CheckItem>
  <CheckItem>
    Service discovery is backed by validated advertisements, freshness handling, and registry persistence.
  </CheckItem>
  <CheckItem>
    The planner produces primary and fallback assignments rather than a single blind dispatch target.
  </CheckItem>
  <CheckItem>
    Runtime execution records fragment-level events that can be surfaced before the job fully completes.
  </CheckItem>
  <CheckItem>
    The final result is not just a status flag. It includes a Qiskit-backed analysis payload suitable for debugging and experiments.
  </CheckItem>
  <CheckItem>
    A separate frontend dashboard exposes the same execution story through topology views, DAG inspection, and result visualizations.
  </CheckItem>
</Checklist>

## Important truths before you rely on a behavior

<DefinitionGrid>
  <Definition term="Planner realism">
    Candidate scoring already incorporates advertised fidelity, but some latency and load signals are still deterministic pseudo-metrics. That keeps the current proof of concept reproducible while leaving room for later telemetry-driven experiments.
  </Definition>
  <Definition term="Distributed versus local">
    Service discovery and remote invocation are genuinely networked through `py-libp2p`, but the coordinator, SQLite persistence, and Qiskit reconstruction all still run in a single local process.
  </Definition>
  <Definition term="Durability scope">
    Jobs, reservations, runtime events, and registry snapshots are persisted. Compiled plans are still cached in memory, which means old `plan_id` values are not durable across coordinator restarts.
  </Definition>
  <Definition term="Quantum semantics">
    Some domain services such as `syndrome_extraction` and `distillation` are represented today as orchestration-level steps rather than fully modeled unitary evolution in the Qiskit reconstruction path.
  </Definition>
</DefinitionGrid>
