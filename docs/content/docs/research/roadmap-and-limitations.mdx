<Spotlight eyebrow="Current Maturity" title="Promising, inspectable, and still intentionally incomplete">
The coordinator already has enough structure to support serious local experiments and convincing demos. It does not yet have every capability promised by the earlier design and requirements documents, and that gap is worth stating plainly.
</Spotlight>

## Major capability areas that are already in place

- API-driven job lifecycle
- registry-backed planning input
- deterministic fragment assignment with fallbacks
- reservation state handling
- runtime retry and degradation handling
- SQLite-backed persistence for key operational records
- Qiskit-backed result analysis
- a frontend dashboard for visualization and inspection

## Major capability areas still open

<SignalGrid>
  <SignalCard eyebrow="Evaluation" title="Centralized baseline mode">
    The requirements and task plan both call for a comparison path against centralized orchestration. That baseline is still not implemented as a first-class execution mode.
  </SignalCard>
  <SignalCard eyebrow="Experiments" title="Scenario runner and report generation">
    There is not yet a one-command experiment harness that runs scenario matrices, records outcomes, and emits reproducible comparison artifacts.
  </SignalCard>
  <SignalCard eyebrow="Performance" title="Scale and hardening work">
    Published targets for larger topologies and fragment counts are not yet backed by a dedicated benchmarking and tuning pass.
  </SignalCard>
  <SignalCard eyebrow="Control Plane" title="Multi-coordinator behavior">
    The project still assumes one active coordinator. Consensus, leader election, and failover are explicitly out of scope for the current proof of concept.
  </SignalCard>
  <SignalCard eyebrow="Security" title="POC-level protection only">
    Optional API key auth and in-memory rate limiting are useful for demos, but they are not a full production security model.
  </SignalCard>
  <SignalCard eyebrow="Language" title="Limited DSL surface">
    The circuit parser is deliberately narrow. It does not attempt complete OpenQASM semantics or arbitrary custom-gate workflows.
  </SignalCard>
</SignalGrid>

## Limitation categories to keep in mind

### Modeling limitations

- some planner costs are still synthetic
- some domain services are represented as logical protocol steps rather than full unitary evolution
- the fidelity summary still includes placeholder-style semantics in places

### Durability limitations

- plan inspection is not durable across backend restarts
- long-term metrics history is not yet modeled
- SQLite is local process storage, not distributed coordination state

### Operational limitations

- the topology is still easiest to exercise through embedded demo services
- runtime behavior is meaningful, but not yet benchmarked at the scale envisioned in the planning docs
- the project has not yet crossed into an operator-ready deployment model

## Highest-value next steps

1. Implement the centralized baseline mode so distributed-versus-centralized comparisons become concrete.
2. Build an experiment harness that records configuration, seed, runtime outcomes, and analysis artifacts.
3. Strengthen observability around planner decisions and runtime retries.
4. Decide which in-memory inspection artifacts should become durable.
5. Measure behavior under larger topologies and more complex fragment graphs.

## How to read the codebase with these gaps in mind

- treat the current system as an honest and structured POC, not an unfinished toy
- use the existing persistence and analysis surfaces to make problems measurable
- view the missing experiment layer as the most obvious bridge to publishable comparisons
