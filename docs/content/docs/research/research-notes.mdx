<Spotlight eyebrow="Research Positioning" title="A distributed-systems prototype with a quantum-service lens">
This codebase makes the most sense when read as an orchestration and systems experiment. It is exploring what happens when remote quantum-style capabilities are surfaced as discoverable services with fidelity, availability, and routing concerns, rather than as hardcoded local procedures.
</Spotlight>

## Why it is research-relevant

The project already gives you a meaningful surface for experiments around:

- distributed coordination over `py-libp2p`
- service discovery and topology freshness
- deterministic fragment assignment under fidelity constraints
- retry and fallback behavior under degraded execution
- durable job lifecycle and runtime event capture
- post-execution quantum analysis suitable for comparison across runs

## Questions the current implementation can help you explore

<SignalGrid>
  <SignalCard eyebrow="Topology" title="How does freshness shape routing?">
    Because the planner depends on an availability-filtered registry snapshot, you can reason concretely about the effect of stale advertisements, restart invalidation, and fidelity thresholds on planner feasibility.
  </SignalCard>
  <SignalCard eyebrow="Resilience" title="What happens when execution degrades?">
    The runtime already models timeouts, node drops, rejected execution, low observed fidelity, retries, and fallback. That makes resilience behavior visible instead of hypothetical.
  </SignalCard>
  <SignalCard eyebrow="Explainability" title="How inspectable is the whole run?">
    The combination of job records, fragment events, plan inspection, and Qiskit output gives you a fairly rich narrative from input circuit to final result payload.
  </SignalCard>
</SignalGrid>

## What exists today

- FastAPI job API with submission, polling, plan inspection, services, fidelity metrics, and WebSocket updates
- SQLite persistence for jobs, runtime events, reservations, and service registry snapshots
- libp2p-backed discovery and remote invocation path
- deterministic planner with primary and fallback candidates
- reservation conflict handling
- Qiskit-backed output including fidelity and state summaries
- unit and integration tests across planner, registry, runtime, and API flows

## What is not finished yet

- a centralized baseline mode for A/B comparison
- a first-class experiment harness that records comparison runs
- performance and scale benchmarking for publishable workload targets
- multi-coordinator consensus or failover
- production-grade security and tenancy

<Callout type="warning" title="Interpret the current metrics carefully">
The planner already uses real advertised fidelity, but latency and load inputs are still deterministic pseudo-signals. That keeps the system reproducible while leaving room for later phases that ingest real telemetry.
</Callout>

## Practical limitations to remember

- The plan endpoint is not durable across coordinator restarts because plans live in process memory.
- Embedded demo nodes are useful for local experimentation, but they are still part of a controlled environment.
- The API's fidelity endpoint exposes a current aggregate snapshot, not a historical series.
- The parser supports a targeted OpenQASM-like subset rather than the entire language.

## Where the roadmap points next

The implementation plan in `backend/docs/tasks.md` makes the unfinished areas explicit:

- centralized orchestration baseline
- experiment runner and report generation
- controlled fault injection
- performance hardening
- operator-focused runbooks

## What this project is a strong basis for

<Checklist>
  <CheckItem>
    Demonstrating how distributed discovery, scheduling, and fallback could matter for remote quantum-service workflows.
  </CheckItem>
  <CheckItem>
    Building better observability or comparison tooling around coordination outcomes.
  </CheckItem>
  <CheckItem>
    Evolving planner heuristics, runtime policy, and experiment harnesses in a codebase that already has operational structure.
  </CheckItem>
  <CheckItem>
    Teaching the intersection of orchestration systems and quantum-workflow abstractions.
  </CheckItem>
</Checklist>

That makes this repository a good fit for:

- systems prototypes
- coordination-policy experiments
- instrumentation work
- docs and developer-experience improvements

It is a weaker fit for:

- claims about production quantum networking
- fully general OpenQASM compilation
- security-sensitive or multi-tenant deployment scenarios
