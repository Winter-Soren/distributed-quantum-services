<Spotlight eyebrow="Topology Layer" title="Discovery is the planner's input, not a side detail">
The coordinator only makes sensible routing decisions if it has a trustworthy view of what services exist, what they claim to support, how fresh those claims are, and whether those services are currently available. That is the job of the discovery-plus-registry path.
</Spotlight>

## Advertisement schema

Each service advertisement includes:

- `protocol_version`
- `node_id`
- `listen_addrs`
- `service_type`
- `fidelity`
- `qubit_min`
- `qubit_max`
- `availability`
- `updated_at`

Validation rules include:

- protocol version must match the supported version
- `fidelity` must stay within `0.0` through `1.0`
- `qubit_max` must be greater than or equal to `qubit_min`

Invalid payloads are rejected before they ever influence planner decisions.

## Registry identity model

Internally, the registry stores entries by:

```text
(node_id, service_type)
```

That means one node can advertise multiple capabilities, each tracked separately.

## Freshness behavior

The registry is freshness-aware rather than append-only.

It can:

- upsert a new advertisement
- load a saved snapshot from storage
- mark stale entries unavailable after a timeout
- mark all entries unavailable on coordinator restart until they re-advertise
- query entries by service type, minimum fidelity, and availability

## Why restart invalidation matters

Cached advertisements can outlive the peers that produced them. To avoid routing work to ghosts, the coordinator marks cached entries unavailable on startup when libp2p is enabled and waits for fresh advertisements to arrive again.

That is a small but important operational safeguard.

## Local demo behavior

In local demo mode, the coordinator fabric seeds the registry directly with the embedded service advertisements so the local system does not need a noisy startup burst just to become useful.

After that, embedded services continue publishing advertisements on a staggered schedule.

## Query semantics

The planner typically queries the registry with:

- a required `service_type`
- a minimum fidelity threshold
- `available_only=True`

The API surface uses the registry differently:

- `/api/v1/services` returns all known entries, even unavailable ones
- `/api/v1/metrics/fidelity/{node_id}` aggregates current advertisement samples for one node

## What gets persisted

When a registry store is configured, every upserted or availability-modified advertisement is written to SQLite. On startup, the registry reloads that snapshot before live discovery resumes.

This gives the coordinator:

- continuity of topology knowledge across restarts
- a safer base for stale-entry invalidation
- a better debugging story than pure in-memory pubsub state

## Caveats to remember

<DefinitionGrid>
  <Definition term="Freshness is timeout-based">
    The registry uses timestamp age to decide staleness. It does not currently maintain a richer liveness protocol with explicit coordinator consensus or multi-source health scoring.
  </Definition>
  <Definition term="Metrics endpoint scope">
    The fidelity metrics endpoint reflects the current advertisement set for a node, not a historical time-series observability backend.
  </Definition>
  <Definition term="Availability is mutable">
    A service can still exist in the registry while being marked unavailable. Clients should not treat presence in `/services` as proof of schedulability.
  </Definition>
  <Definition term="Planner dependence">
    Every fragment assignment decision begins here. A stale or incomplete registry view directly degrades plan quality and feasibility.
  </Definition>
</DefinitionGrid>
