<Spotlight eyebrow="Architectural Lens" title="A coordinator for distributed quantum-service workflows">
The system is built around a simple but ambitious idea: quantum capabilities should be treated as discoverable services with operational properties, not just as local library calls hidden behind a single backend function. Once you make that shift, planning, reservation, recovery, topology freshness, and observability become first-class concerns.
</Spotlight>

## The three connected planes

<LayerGrid>
  <LayerCard eyebrow="Control Plane" title="Decide what should happen">
    The control plane receives circuits, normalizes them into internal representations, derives dependencies, scores candidate nodes, manages lifecycle transitions, and decides when fragments are eligible for execution.
  </LayerCard>
  <LayerCard eyebrow="Execution Plane" title="Make the work actually happen">
    The execution plane relies on pubsub advertisements, remote gate invocation streams, reservation windows, timeout handling, retries, and fallback routing to move a job from plan to fragment outcomes.
  </LayerCard>
  <LayerCard eyebrow="Data Plane" title="Preserve the operational trail">
    The data plane captures jobs, registry snapshots, reservations, runtime events, and final analysis payloads so that runs are inspectable and partially recoverable after process restarts.
  </LayerCard>
</LayerGrid>

## End-to-end flow

<Timeline>
  <TimelineItem eyebrow="1. Submit" title="A client sends circuit text">
    The entry point is `POST /api/v1/circuits/submit`. The API validates the request, enforces optional auth and rate-limiting policy, persists the job as `QUEUED`, and schedules background processing.
  </TimelineItem>
  <TimelineItem eyebrow="2. Compile" title="The planner turns text into a routed plan">
    `JobManager` hands the circuit to the planner, which normalizes the language, builds dependency structure, fragments the workflow, and assigns primary plus fallback node candidates.
  </TimelineItem>
  <TimelineItem eyebrow="3. Reserve" title="The runtime asks for an execution window">
    Before invoking a fragment, the reservation protocol checks whether the target node is still eligible, available, and conflict-free for the requested service type and time window.
  </TimelineItem>
  <TimelineItem eyebrow="4. Execute" title="The runtime invokes fragment work">
    Ready fragments execute in dependency-safe order. Timeouts, node drops, rejected invocations, and low observed fidelity can trigger retries or fallback routing.
  </TimelineItem>
  <TimelineItem eyebrow="5. Persist" title="Operational state is recorded continuously">
    Job transitions, reservation state, registry snapshots, and fragment attempt events are written to SQLite so that the system remains inspectable even across process restarts.
  </TimelineItem>
  <TimelineItem eyebrow="6. Analyze" title="Qiskit reconstructs the quantum view">
    Once all fragments succeed, the backend builds counts, probabilities, statevector summaries, Bloch vectors, entropy, and fidelity information for downstream consumers.
  </TimelineItem>
</Timeline>

## Key backend modules

| Module | Responsibility |
| --- | --- |
| `backend/src/quantum_coordinator/api/app.py` | FastAPI assembly, endpoints, auth/rate-limit policy, startup and shutdown hooks |
| `backend/src/quantum_coordinator/application/job_manager.py` | Job submission, lifecycle transitions, plan cache, progress tracking, and recovery |
| `backend/src/quantum_coordinator/planning/` | Circuit normalization, dependency DAG construction, fragment creation, and candidate scoring |
| `backend/src/quantum_coordinator/reservation/` | Reservation request handling and state transitions |
| `backend/src/quantum_coordinator/runtime/` | Runtime scheduling, retry logic, fallback handling, and Qiskit result building |
| `backend/src/quantum_coordinator/service_discovery/` | Advertisement validation, registry freshness, and availability tracking |
| `backend/src/quantum_coordinator/infra/persistence/` | SQLite storage for jobs, runtime events, reservations, and registry state |
| `backend/src/quantum_coordinator/infra/libp2p/` | Coordinator fabric, protocols, and gate invocation adapter |

## One practical topology to keep in mind

For local demo runs, the system typically looks like this:

```text
client or dashboard
        |
        v
   FastAPI coordinator
        |
        +--> JobManager
        +--> CircuitPlanner
        +--> ReservationProtocol
        +--> RuntimeExecutor
        +--> SQLite stores
        |
        v
   py-libp2p fabric
        |
        +--> embedded service node A
        +--> embedded service node B
        +--> embedded service node C
```

This topology is still valuable because it exercises the real transport and advertisement path while remaining runnable on a single machine.

## Startup and recovery behavior

On startup the coordinator:

- runs SQLite migrations
- restores the service registry snapshot
- prunes stale advertisements
- marks cached libp2p services unavailable until they re-advertise
- starts the libp2p fabric when enabled
- retries unfinished jobs from persistent storage

That recovery path is one of the more important differences between this project and a toy simulator: jobs and registry state are treated as operational state, not ephemeral demo data.

## What is distributed versus local today

<DefinitionGrid>
  <Definition term="Distributed today">
    Service advertisement, discovery ingestion, remote invocation, and coordinator-side decision making happen through the libp2p-driven network path.
  </Definition>
  <Definition term="Local today">
    SQLite persistence, the coordinator process, and the Qiskit reconstruction stage remain local to the single running application.
  </Definition>
  <Definition term="Not implemented yet">
    There is no multi-coordinator control plane, no consensus, no leader election, and no cross-coordinator failover path.
  </Definition>
  <Definition term="Why that still matters">
    Even within one process, the code already models operational boundaries that production systems would need: discovery freshness, retries, fallback, persistence, and clear API contracts.
  </Definition>
</DefinitionGrid>
