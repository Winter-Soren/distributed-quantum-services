<Spotlight eyebrow="Analysis Layer" title="The final payload is a reconstruction, not a hardware trace">
After runtime execution succeeds, the backend uses Qiskit to rebuild an interpreted quantum view of the routed workflow. That gives users more than a binary success flag, but it also comes with important modeling assumptions that are worth understanding.
</Spotlight>

## What gets returned

The `quantum_result` payload can include:

- `counts`
- `probabilities`
- `measured_probabilities`
- `statevector`
- `shots`
- `measured_qubits`
- `observable_expectations`
- `reduced_density_matrices`
- `bloch_vectors`
- `entanglement_entropy`
- `fidelity`
- `top_basis_states`

## How the reconstruction works

The result builder:

1. infers the total qubit count from the plan
2. rebuilds a Qiskit circuit in fragment order
3. applies service-specific or raw-operation-specific mappings
4. computes the statevector from the reconstructed circuit
5. samples counts when measurement operations are present
6. derives observables, reduced states, Bloch vectors, entropy, and fidelity summaries

## Important service mappings

Some services map directly to unitary operations:

- `hadamard` -> `H`
- `bell_pair` -> `H` then `CX`
- `cnot` -> `CX`
- `cz` -> `CZ`
- `qft` -> `QFTGate`

Some services are interpreted more pragmatically:

- `teleportation` -> modeled as `SWAP`
- `measurement_feedforward` -> marks qubits as measured
- `syndrome_extraction` and `distillation` -> treated as logical orchestration steps without extra unitary evolution

## Why that distinction matters

The analysis payload is excellent for:

- sanity-checking routed workflows
- comparing observed fragment fidelity against idealized compiled state behavior
- building an explainable debugging story around the workflow

It is weaker as evidence for:

- detailed physical modeling of every domain-specific protocol step
- claims about actual remote hardware state evolution
- exact interpretation of ancilla-heavy or protocol-level workflows not fully encoded in the DSL

## Counts versus probabilities versus statevector

<DefinitionGrid>
  <Definition term="Counts">
    Sampled readout counts over measured qubits. These only appear when measurement steps exist.
  </Definition>
  <Definition term="Measured probabilities">
    The ideal probability distribution over the measured qubits before sampling noise is introduced.
  </Definition>
  <Definition term="Probabilities">
    The ideal probability distribution over the full reconstructed state.
  </Definition>
  <Definition term="Statevector">
    The pre-measurement amplitude view of the reconstructed circuit state.
  </Definition>
</DefinitionGrid>

## Derived analysis fields

### Observable expectations

The backend computes:

- single-qubit `Z` expectations
- pairwise `ZZ` expectations
- pairwise `XX` expectations

### Reduced density matrices

For each qubit, the backend computes the single-qubit reduced density matrix by tracing out all other qubits.

### Bloch vectors

Each reduced single-qubit state is summarized as `x`, `y`, and `z` coordinates in Bloch space.

### Entanglement entropy

For each qubit-versus-rest partition, the backend computes von Neumann entropy in base 2.

### Top basis states

The payload also includes the highest-probability basis states with amplitudes so that users do not have to parse the full statevector first.

## Fidelity summary caveat

The current fidelity summary includes:

- `target_state: "ideal_compiled_state"`
- `fidelity_to_target_state`
- `estimated_execution_fidelity`

The `estimated_execution_fidelity` is built from the product of observed fragment fidelities. The `fidelity_to_target_state` is currently computed by comparing the reconstructed statevector to itself, which makes it a structural placeholder rather than a discriminating external benchmark.

That does not make the field useless, but it does mean you should interpret it as part of an evolving research surface rather than a finished fidelity-assessment framework.

## Best way to use this payload today

- treat it as a rich explanation and debugging surface
- compare it with fragment execution metadata
- use it to reason about orchestration outcomes
- avoid overselling it as a perfect physical model of all service-level semantics
