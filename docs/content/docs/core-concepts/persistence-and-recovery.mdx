<Spotlight eyebrow="Durability Model" title="Operational state is persisted, but not every convenience artifact">
The coordinator deliberately persists the pieces of state that matter for observability and restart recovery: jobs, registry snapshots, reservations, and fragment execution events. At the same time, some inspection helpers such as the compiled plan cache remain in memory only. Understanding that split is important when debugging restarts.
</Spotlight>

## SQLite tables created by migrations

The initial schema migration creates:

- `jobs`
- `service_ads`
- `reservations`
- `fragment_execution_events`
- `schema_migrations`

Each table exists to support a different operational question:

- `jobs`: what is the latest known lifecycle state of a workflow?
- `service_ads`: what topology snapshot did we last know about?
- `reservations`: which execution windows were requested, committed, rejected, or canceled?
- `fragment_execution_events`: what actually happened attempt by attempt during runtime?

## Persistence adapters in the codebase

| Store | Responsibility |
| --- | --- |
| `SQLiteJobStore` | current job record and unfinished-job listing |
| `SQLiteServiceRegistryStore` | registry snapshot persistence |
| `SQLiteRuntimeEventStore` | reservation records plus fragment execution events |

## What is durable

<Checklist>
  <CheckItem>
    A job is persisted immediately when it is submitted, before background processing starts.
  </CheckItem>
  <CheckItem>
    Reservation transitions are written as the protocol progresses.
  </CheckItem>
  <CheckItem>
    Fragment attempt events are appended in execution order for later progress reconstruction.
  </CheckItem>
  <CheckItem>
    Service advertisements and availability changes are saved as the registry view changes.
  </CheckItem>
</Checklist>

## What is not fully durable yet

- compiled execution plans in `JobManager._plans`
- any frontend-side browser session state outside the browser
- a long-lived fidelity-history model beyond the current snapshot

This is why an old `job_id` can still resolve while the corresponding `plan_id` no longer does after a restart.

## Recovery behavior on startup

The application startup sequence does a few important things in order:

1. runs migrations
2. reconstructs the service registry from SQLite
3. prunes stale advertisements
4. marks cached services unavailable until fresh advertisements arrive when libp2p is enabled
5. reloads unfinished jobs and attempts to process them

That means recovery is intentionally operational rather than cosmetic.

## What "unfinished job recovery" really means

The job store treats these statuses as unfinished:

- `QUEUED`
- `COMPILING`
- `RESERVING`
- `EXECUTING`

On startup the coordinator iterates those jobs and asks `JobManager` to process them again.

That gives the system a simple but meaningful restart story:

- durable job records are not abandoned
- the application can continue work that was in flight
- duplicate concurrent processing is avoided with the in-memory claim guard

## Caveat: WAL configuration

The configuration model includes `database.enable_wal_mode`, but the current SQLite store implementations do not actively apply a WAL pragma. That means the setting exists conceptually, but the actual storage code is not yet wiring it through.

This is exactly the kind of detail these docs try to expose honestly.

## Database path and environment control

By default, the database path is:

```text
./data/quantum_coordinator.db
```

You can override it via config or environment variables. See [Configuration](../reference/configuration.mdx) for the full shape.

## Operational implications

<DefinitionGrid>
  <Definition term="Good for inspection">
    SQLite is perfectly adequate for local research runs, demos, and reproducible backend debugging.
  </Definition>
  <Definition term="Not a distributed source of truth">
    The current design does not claim multi-process transactional orchestration across several coordinators.
  </Definition>
  <Definition term="Enough for recovery experiments">
    Because jobs and runtime events are durable, restart behavior is observable and testable rather than hand-waved.
  </Definition>
  <Definition term="Still evolving">
    The persistence layer is one of the best candidates for future hardening work, especially around richer metrics history and stronger durability guarantees for plan inspection.
  </Definition>
</DefinitionGrid>
