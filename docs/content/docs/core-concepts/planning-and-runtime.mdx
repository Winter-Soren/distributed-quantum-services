<Spotlight eyebrow="Critical Path" title="From circuit text to runtime outcome">
This is the most important path in the system. It is where language handling, topology awareness, scheduling policy, retry logic, and analysis all meet. If you understand this page, you understand the coordinator's real behavior.
</Spotlight>

## 1. Normalization

The planner starts by converting OpenQASM-like input into an internal circuit representation.

During normalization, the backend:

- strips `//` comments
- detects OpenQASM 2 versus OpenQASM 3 style declarations
- extracts qubit count from `qreg` or `qubit[...]`
- maps supported gates and aliases onto internal service types
- expands supported `for i in [start:end] { ... }` loops

If the parser cannot recognize the circuit shape, planning fails early with a descriptive error.

## 2. Dependency and fragment derivation

Once operations are normalized, the backend builds operation dependencies and then groups work into execution fragments. The planner ultimately cares about fragment dependencies, not just raw instruction order, because runtime scheduling only needs to respect the edges that matter.

Each fragment carries:

- `fragment_id`
- `service_type`
- `qubits`
- `operation_ids`
- `dependencies`
- `raw_text`

That same structure later appears in `GET /api/v1/plans/{plan_id}`.

## 3. Scheduling order

The planner then computes a topological fragment order. That order matters because the runtime walks it repeatedly to find fragments whose dependencies have all completed successfully.

## 4. Candidate scoring

For every fragment, the planner queries the registry for available services that satisfy the minimum fidelity threshold.

The current planner configuration defaults are:

- minimum fidelity: `0.80`
- fallback count: `2`
- weights: `w_lat=0.25`, `w_fail=0.50`, `w_ent=0.15`, `w_load=0.10`

The total score follows this shape:

```text
total_cost =
  w_lat * latency_cost +
  w_fail * failure_risk_cost +
  w_ent * entanglement_cost +
  w_load * load_cost
```

<Callout type="warning" title="Current scoring is partly synthetic">
In the current proof of concept, `latency_cost` and `load_cost` are deterministic pseudo-metrics rather than live network telemetry. Fidelity comes from real advertisements, but the rest of the scoring model is still a controlled stand-in for later experimental phases.
</Callout>

The planner sorts candidates by total cost, then by higher fidelity, then by node id. The top candidate becomes the primary assignment and the next two become fallbacks.

## 5. Reservation state and node selection

Before a fragment can execute on a chosen node, the runtime asks the reservation protocol for an execution window.

The current reservation state machine is:

```text
REQUESTED -> PREPARED -> COMMITTED -> EXECUTED
                             |-> EXPIRED
                             |-> CANCELED
REQUESTED/PREPARED --------> REJECTED
```

Reservations are rejected when:

- the node is no longer eligible for the requested service and fidelity
- an overlapping committed reservation already exists for that node and service type

## 6. Runtime execution flow

At runtime, the executor walks the fragment order and only schedules fragments whose dependencies have already succeeded.

For each node candidate, it:

1. creates a reservation window
2. checks node eligibility and reservation conflicts
3. invokes the fragment through the gate execution adapter
4. retries on transient failure up to the configured maximum
5. falls back to the next node when the current one cannot complete successfully

The runtime policy defaults include:

- `max_retries = 3`
- `timeout_seconds = 5.0`
- bounded exponential backoff with jitter
- `min_fidelity = 0.8`

## 7. Fragment event model

The runtime store records fragment events such as:

- `SUCCESS`
- `TIMEOUT`
- `NODE_DROP`
- `REJECTED`
- `QUALITY_DEGRADED`

These events drive:

- live job progress
- the dashboard status rail
- the fragment timeline shown before final completion

## 8. Job lifecycle

At the job level, the API exposes this state progression:

```text
QUEUED -> COMPILING -> RESERVING -> EXECUTING -> COMPLETED | FAILED
```

The `JobManager` persists each transition and guards against duplicate in-flight processing.

## 9. Final quantum result

After all fragments succeed, Qiskit reconstructs a quantum result payload that can include:

- `counts`
- `probabilities`
- `measured_probabilities`
- `statevector`
- `observable_expectations`
- `reduced_density_matrices`
- `bloch_vectors`
- `entanglement_entropy`
- `fidelity`
- `top_basis_states`

The backend treats those as **post-execution analysis artifacts**, not as evidence that a real remote quantum device was simulated end to end.

## 10. Failure matrix

<DefinitionGrid>
  <Definition term="Planning failure">
    Happens before execution begins. Typical causes are unsupported circuit syntax or missing feasible services in the registry.
  </Definition>
  <Definition term="Reservation failure">
    Happens when a candidate node is unavailable, below the minimum fidelity threshold, or already reserved for the same service window.
  </Definition>
  <Definition term="Execution failure">
    Happens when a fragment times out, a node drops, the invocation is rejected, or the observed fidelity comes back too low.
  </Definition>
  <Definition term="Terminal job failure">
    Happens when the runtime exhausts retries and fallbacks for a required fragment. The final job record captures the error text.
  </Definition>
</DefinitionGrid>

## Failure behavior to keep in mind

- If no node satisfies a fragment's capability or fidelity requirement, planning fails before execution begins.
- If a fragment exhausts retries and fallbacks, the job fails terminally with the last fragment-level error.
- Progress reporting depends on the compiled plan being present in the in-memory plan cache for the current process.
