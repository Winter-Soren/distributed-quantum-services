<Spotlight eyebrow="Language Scope" title="A pragmatic OpenQASM-like DSL">
The coordinator accepts circuit text that looks like OpenQASM 2 or OpenQASM 3, but it does not claim full language coverage. The goal is to support the workflows needed by the planner and runtime, not to implement a universal compiler frontend.
</Spotlight>

## Supported declarations

The parser recognizes either style:

```txt
OPENQASM 2.0;
qreg q[2];
creg c[1];
```

```txt
OPENQASM 3;
qubit[2] q;
bit[1] c;
```

`include`, `creg`, and `bit` lines are tolerated, but the planner mainly cares about the qubit register declaration and executable operations.

## Supported gate mapping

| Input text | Normalized service type |
| --- | --- |
| `h`, `hadamard` | `hadamard` |
| `cx`, `cnot` | `cnot` |
| `cz` | `cz` |
| `controlled_u`, `controlled <gate>` | `controlled_unitary` |
| `qft` | `qft` |
| `teleport`, `teleportation` | `teleportation` |
| `bell`, `bell_pair` | `bell_pair` |
| `syndrome_extraction` | `syndrome_extraction` |
| `distillation` | `distillation` |
| `measure`, `measurement_feedforward` | `measurement_feedforward` |

Any other recognized gate-shaped line is normalized to `programmable_gate`.

## Raw-operation fallback behavior

Even after normalization, the result-analysis path may inspect the original raw operation text to apply a more specific Qiskit reconstruction when possible. That is why several gate-like inputs can still produce meaningful result analysis even when they are represented internally as broader service categories.

## Qubit references

The parser understands:

- single indices such as `q[0]`
- ranges such as `q[0:2]`
- arithmetic expressions inside indices and loop bounds

Supported arithmetic operators include:

- `+`
- `-`
- `*`
- `/`
- `//`
- `%`
- `^` (normalized internally to exponentiation)

Expressions must still resolve to integer-valued qubit indices or loop bounds. Negative indices are rejected.

## Supported loop form

The parser expands this loop syntax before planning:

```txt
for i in [0:2] {
  cx q[i], q[i + 1];
}
```

Nested loops are supported as long as the syntax stays within this restricted pattern.

## Measurement handling

Measurement lines are treated as `measurement_feedforward` operations. For a statement like:

```txt
measure q[0] -> c[0];
```

the parser extracts the qubit side and ignores the classical register side for planning purposes.

## A few examples the parser understands

<DefinitionGrid>
  <Definition term="OpenQASM 2 register style">
    `qreg q[2];` is accepted and used to infer total qubit count.
  </Definition>
  <Definition term="OpenQASM 3 register style">
    `qubit[2] q;` is also accepted and marks the input as OpenQASM 3 style.
  </Definition>
  <Definition term="Alias normalization">
    `cx` and `cnot` normalize to the same service type. `teleport` and `teleportation` do too.
  </Definition>
  <Definition term="Programmable fallback">
    Unrecognized gate-shaped operations default to `programmable_gate` rather than failing instantly, which keeps the planner flexible while preserving the original raw text.
  </Definition>
</DefinitionGrid>

## A realistic example

```txt
OPENQASM 3;
qubit[2] q;
bit[1] c;

bell_pair q[0], q[1];
cnot q[0], q[1];
cz q[0], q[1];
teleport q[0], q[1];
syndrome_extraction q[0];
distillation q[1];
measure q[0] -> c[0];
```

## Current limitations

- This is not a full OpenQASM parser.
- Custom gate definitions are not compiled as reusable language constructs.
- Classical control flow beyond the restricted `for ... in [start:end]` form is not supported.
- Advanced include semantics are ignored.
- Parser acceptance does not guarantee a semantically rich Qiskit reconstruction for every custom operation.

If you need to know how this normalized circuit becomes fragment assignments and fallback candidates, read [Planning and Runtime](../core-concepts/planning-and-runtime.mdx).
