<Spotlight eyebrow="Interface Layer" title="These contracts connect the coordinator's moving parts">
The system is easier to extend and debug when its message shapes are explicit. This page documents the main contracts that tie discovery, reservation, and execution together, whether they travel over the network or only between backend components.
</Spotlight>

## 1. Service advertisement

The advertisement schema is a validated Pydantic model and represents the coordinator's discoverable view of each service capability.

| Field | Meaning |
| --- | --- |
| `protocol_version` | schema compatibility version |
| `node_id` | unique libp2p peer identity |
| `listen_addrs` | reachable multiaddrs for the node |
| `service_type` | normalized gate or protocol capability |
| `fidelity` | advertised quality value from `0.0` to `1.0` |
| `qubit_min` | minimum qubit support |
| `qubit_max` | maximum qubit support |
| `availability` | whether the service is currently schedulable |
| `updated_at` | freshness timestamp |

Validation rules:

- protocol version must match the supported version
- qubit range must be valid
- fidelity must remain in bounds

## 2. Reservation request and response

The reservation layer works with explicit dataclass contracts.

### Reservation request

| Field | Meaning |
| --- | --- |
| `reservation_id` | generated request identifier |
| `job_id` | owning job |
| `fragment_id` | fragment being scheduled |
| `node_id` | requested execution target |
| `service_type` | requested capability |
| `min_fidelity` | minimum acceptable fidelity |
| `window_start` | start of the requested execution window |
| `window_end` | end of the requested execution window |
| `created_at` | request creation time |

### Reservation response

| Field | Meaning |
| --- | --- |
| `request_id` | echoes the reservation request id |
| `accepted` | whether the reservation succeeded |
| `state` | resulting reservation state |
| `reservation_id` | committed reservation id when accepted |
| `reason` | rejection or cancellation explanation |
| `suggested_window_start` | optional conflict-aware alternative start time |

Typical rejection reasons include:

- `unavailable_or_low_fidelity`
- `reservation_conflict`

## 3. Reservation record lifecycle

The persistence layer stores reservation records with:

- `reservation_id`
- `job_id`
- `fragment_id`
- `node_id`
- `service_type`
- `min_fidelity`
- `window_start`
- `window_end`
- `state`
- `reason`
- `updated_at`

These records let the system reconstruct how scheduling decisions evolved over time.

## 4. Gate execution request over libp2p

The current libp2p gate adapter serializes JSON requests shaped like:

```json
{
  "fragment_id": "frag-0001",
  "service_type": "cnot",
  "qubits": [0, 1],
  "timeout_seconds": 5.0,
  "min_fidelity": 0.8
}
```

This is the minimum information a remote node needs to interpret an invocation in the current proof of concept.

## 5. Gate execution response over libp2p

The adapter expects a JSON response that can be reduced to:

- `success`
- `observed_fidelity`
- `error`

If the response cannot be decoded correctly, the adapter normalizes it to:

```json
{
  "success": false,
  "observed_fidelity": 0.0,
  "error": "invalid_gate_response"
}
```

## 6. Fragment execution event record

Every recorded execution attempt can include:

- `event_id`
- `job_id`
- `fragment_id`
- `node_id`
- `attempt`
- `status`
- `error`
- `observed_fidelity`
- `created_at`

These events are the backbone for progress reconstruction and runtime diagnostics.

## 7. Job update payload over WebSocket

The WebSocket route uses a compact update contract:

- `job_id`
- `status`
- `error`
- `progress`
- `updated_at`

This is intentionally lighter than the full job result model so clients can receive frequent updates without downloading the whole result repeatedly.

## Contract philosophy

<DefinitionGrid>
  <Definition term="Small payloads">
    The current contracts try to carry only what the runtime and UI need. That keeps the proof of concept understandable and easier to test.
  </Definition>
  <Definition term="Validated edges">
    Discovery data is validated at the edge so malformed topology information cannot quietly poison planner decisions.
  </Definition>
  <Definition term="Recoverable records">
    Reservation and execution records are shaped so they can be persisted and inspected after the fact, not just used in memory and discarded.
  </Definition>
  <Definition term="Room to grow">
    These contracts are intentionally simple enough to evolve toward richer telemetry, stronger auth, or more detailed execution semantics later.
  </Definition>
</DefinitionGrid>
