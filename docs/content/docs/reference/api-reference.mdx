<Spotlight eyebrow="HTTP Surface" title="A small API with a lot of state behind it">
The backend intentionally exposes a compact public surface. Most complexity lives behind the endpoints in planning, reservation, runtime, persistence, and analysis layers. This page tells you exactly what each route promises today and what caveats you should keep in mind while consuming it.
</Spotlight>

## Authentication and rate limiting

- API key auth is optional and controlled by config.
- When auth is enabled, send `X-API-Key: <value>` on HTTP requests.
- The WebSocket endpoint uses the same `X-API-Key` header.
- Request rate limiting is optional and applied per client host.

## Status model used across the API

Job status values are:

```text
QUEUED
COMPILING
RESERVING
EXECUTING
COMPLETED
FAILED
```

These values appear in:

- submit responses
- job polling responses
- WebSocket job update payloads
- frontend lifecycle visualization

## Endpoints

| Endpoint | Method | Purpose |
| --- | --- | --- |
| `/api/v1/health` | `GET` | Health, version, environment, and uptime |
| `/api/v1/circuits/submit` | `POST` | Submit a circuit for background execution |
| `/api/v1/jobs/{job_id}` | `GET` | Retrieve current status, progress, and results |
| `/api/v1/plans/{plan_id}` | `GET` | Inspect the compiled execution plan |
| `/api/v1/services` | `GET` | List the current service registry snapshot |
| `/api/v1/metrics/fidelity/{node_id}` | `GET` | Aggregate current fidelity samples for one node |
| `/api/v1/jobs/{job_id}/ws` | `WS` | Stream job status changes until completion |

## `POST /api/v1/circuits/submit`

What it does:

- validates the request body
- enforces payload-size limits
- persists a queued job immediately
- schedules background processing

Request body:

```json
{
  "circuit": "OPENQASM 3;\nqubit[2] q;\nbit[1] c;\ncnot q[0], q[1];\nmeasure q[0] -> c[0];"
}
```

Response shape:

```json
{
  "job_id": "job-...",
  "status": "QUEUED"
}
```

Important behavior:

- the job is persisted immediately
- processing happens in the background
- payload size is capped by `api.max_request_bytes`

Likely error cases:

- `413` when the circuit payload is too large
- `401` when auth is enabled and the API key is missing or invalid
- `429` when rate limiting is enabled and exceeded

## `GET /api/v1/jobs/{job_id}`

Query parameter:

- `result_detail=full|summary`

The response includes:

- `status`
- `plan_id`
- `error`
- `progress`
- `result`
- timestamps for creation and last update

### `result_detail=summary`

The summary mode removes the heaviest quantum fields:

- `probabilities`
- `statevector`
- `reduced_density_matrices`

Use it for dashboards or polling loops where you do not need full quantum detail on every request.

### In-progress jobs

Before completion, the API can still return partial `fragment_results` for fragments that have already succeeded. In that case, `quantum_result` remains `null` until the whole job completes.

### Progress semantics

When a compiled plan is available in the current process, the response can also include:

- `total_fragments`
- `completed_fragments`
- `active_fragments`
- `completion_ratio`
- `latest_event_at`
- `finalizing`

This is derived from the plan cache plus persisted runtime events.

### Typical status-code behavior

- `404` if the job is unknown
- `200` with partial data for in-progress jobs
- `200` with final `result` for completed jobs

## `GET /api/v1/plans/{plan_id}`

Plan responses include:

- `fragment_order`
- per-fragment dependency data
- `primary_node_id`
- `fallback_node_ids`
- candidate cost breakdown
- `quality_snapshot_id`

<Callout type="warning" title="Plan lookup is currently session-scoped">
Compiled plans are stored in the coordinator's in-memory cache. That means plan inspection works for plans compiled since the current process started, but old `plan_id` values are not durable across coordinator restarts.
</Callout>

Use this endpoint when you want to understand:

- why a fragment went to a specific node
- what fallback nodes existed
- how many fragments the planner derived
- what candidate costs looked like at compile time

## `GET /api/v1/services`

This endpoint returns the coordinator's current registry snapshot for all known advertisements, not just available ones. Clients should always inspect the `availability` field rather than assuming every returned service can accept work.

Each item includes:

- `node_id`
- `listen_addrs`
- `service_type`
- `fidelity`
- `qubit_min`
- `qubit_max`
- `availability`
- `updated_at`

This is the best route for answering:

- which services currently exist
- which nodes are advertising multiple capabilities
- whether stale services are still present but marked unavailable
- how the frontend should construct a topology summary

## `GET /api/v1/metrics/fidelity/{node_id}`

Despite the endpoint name, the current implementation returns a **current snapshot aggregate** for a node rather than a historical time series.

The payload includes:

- `sample_count`
- `average_fidelity`
- `min_fidelity`
- `max_fidelity`
- one sample per currently advertised service type

If the node is unknown to the current registry, the endpoint returns `404`.

This endpoint is most useful for:

- node summary panels in the frontend
- quick local inspection of capability quality
- validating that a node advertises multiple service types at different fidelities

## `WS /api/v1/jobs/{job_id}/ws`

The WebSocket sends JSON payloads shaped like:

```json
{
  "job_id": "job-...",
  "status": "EXECUTING",
  "error": null,
  "progress": {
    "total_fragments": 6,
    "completed_fragments": 3,
    "active_fragments": 1,
    "completion_ratio": 0.5,
    "latest_event_at": "2026-03-17T12:34:56Z",
    "finalizing": false
  },
  "updated_at": "2026-03-17T12:34:56Z"
}
```

Behavior to know:

- the socket only emits when the payload changes
- it closes normally when the job reaches `COMPLETED` or `FAILED`
- it closes with `4404` if the job does not exist
- auth or rate-limit failures close the socket before acceptance

This is the right interface for UIs that want responsive job status without high-frequency polling.

## Result model highlights

The final `quantum_result` can contain both execution-oriented and analysis-oriented fields:

- `counts` and `measured_probabilities` describe measured outputs
- `probabilities` and `statevector` describe the pre-measurement state
- `bloch_vectors`, `entanglement_entropy`, and `reduced_density_matrices` support deeper analysis
- `fidelity` provides target-state and estimated execution fidelity summaries

## API limitations worth documenting explicitly

<DefinitionGrid>
  <Definition term="Plan durability">
    Plan inspection is tied to the current process because plans are cached in memory rather than persisted as a standalone durable object.
  </Definition>
  <Definition term="Metrics history">
    The fidelity metrics route exposes a current aggregate view, not a long-lived historical telemetry service.
  </Definition>
  <Definition term="Background execution model">
    Job processing starts in background tasks within the API process. The current design is not an external queue worker system.
  </Definition>
  <Definition term="Security posture">
    Auth and rate limiting are useful POC protections, not a full production perimeter.
  </Definition>
</DefinitionGrid>
