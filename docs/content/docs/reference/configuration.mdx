<Spotlight eyebrow="Configuration Model" title="File-first with environment overrides">
The coordinator loads configuration from an optional file and then applies `QC_`-prefixed environment overrides. The config surface is small enough to reason about but rich enough to change runtime behavior, security posture, network mode, and persistence details.
</Spotlight>

The coordinator loads configuration from:

1. an optional config file
2. environment overrides

The environment prefix is `QC_`, and the file-path override variable is `QC_CONFIG_FILE`.

## Supported config file formats

- `.yaml`
- `.yml`
- `.toml`

The repository ships with a baseline file at `backend/config/config.example.yaml`.

## Example YAML

```yaml
environment: development

api:
  host: 0.0.0.0
  port: 8080
  enable_cors: true
  max_request_bytes: 1000000
  enable_auth: false
  enable_rate_limit: false

database:
  path: ./data/quantum_coordinator.db

discovery:
  stale_timeout_seconds: 120
  service_ad_topic: /quantum-coordinator/service-ads/v1

runtime:
  max_retries: 3
  base_timeout_seconds: 5.0
  backoff_multiplier: 2.0

libp2p:
  enabled: true
  coordinator_listen_addrs:
    - /ip4/127.0.0.1/tcp/9100
  embedded_service_count: 3
  embedded_service_base_port: 9200
```

## High-value settings

| Setting | Why it matters |
| --- | --- |
| `api.max_request_bytes` | Caps circuit payload size before processing begins |
| `api.enable_auth` and `api.api_key` | Turns on header-based API protection |
| `api.enable_rate_limit` | Enables fixed-window request limiting |
| `database.path` | Controls where jobs, registry snapshots, and runtime events are stored |
| `discovery.stale_timeout_seconds` | Determines when stale service advertisements are considered unavailable |
| `runtime.max_retries` | Sets fragment-level retry budget |
| `runtime.base_timeout_seconds` | Controls reservation window size and execution timeout baseline |
| `runtime.backoff_multiplier` | Shapes retry backoff growth |
| `libp2p.enabled` | Switches between real libp2p transport and local in-process execution |
| `libp2p.embedded_service_count` | Controls how many demo services are embedded for local runs |

## Configuration sections in plain language

<SignalGrid>
  <SignalCard eyebrow="API" title="HTTP-facing behavior">
    Host, port, CORS policy, payload-size limits, optional API key enforcement, and optional rate limiting all live under the `api` section.
  </SignalCard>
  <SignalCard eyebrow="Discovery" title="Topology freshness">
    Discovery settings control how often advertisements are expected, how quickly they become stale, and which pubsub topic is used for service announcements.
  </SignalCard>
  <SignalCard eyebrow="Runtime" title="Retry and timeout policy">
    The runtime section shapes execution behavior, including how long the system waits, how many times it retries, and how aggressively backoff grows.
  </SignalCard>
  <SignalCard eyebrow="libp2p" title="Transport mode">
    This section controls whether real `py-libp2p` transport is used, which listen addresses the coordinator binds, and how many embedded services are started in local demo runs.
  </SignalCard>
  <SignalCard eyebrow="Logging" title="Operational visibility">
    Logging config determines the service name, level, and whether JSON logs are emitted.
  </SignalCard>
  <SignalCard eyebrow="Database" title="Local state location">
    The database section mainly controls the SQLite path today, plus a WAL flag that is modeled in config but not yet fully wired through the persistence implementation.
  </SignalCard>

## Environment overrides

Nested settings are overridden with `__` separators.

Examples:

```bash
QC_CONFIG_FILE=backend/config/config.example.yaml
QC_API__PORT=9000
QC_LOGGING__LEVEL=DEBUG
QC_API__ENABLE_AUTH=true
QC_API__API_KEY=super-secret
QC_LIBP2P__ENABLED=false
QC_LIBP2P__COORDINATOR_LISTEN_ADDRS='["/ip4/127.0.0.1/tcp/9100"]'
```

## How values are coerced

The loader attempts to coerce environment values in this order:

1. booleans
2. integers
3. floats
4. JSON
5. raw string fallback

That is why list-valued fields such as listen addresses should be passed as JSON strings.

## Example override profiles

### Disable libp2p for fast local API work

```bash
QC_LIBP2P__ENABLED=false
```

### Turn on simple API key auth

```bash
QC_API__ENABLE_AUTH=true
QC_API__API_KEY=super-secret
```

### Change the API port

```bash
QC_API__PORT=9000
```

### Point at a different database path

```bash
QC_DATABASE__PATH=./data/dev-alt.db
```

## Starting with an explicit config file

```bash
cd backend
QC_CONFIG_FILE=config/config.example.yaml uv run uvicorn quantum_coordinator.asgi:app --host 0.0.0.0 --port 8080
```

## Validation behavior

- unsupported file extensions fail fast
- missing config files fail fast
- invalid schemas fail fast with a validation error

That makes configuration problems visible at startup rather than surfacing later during runtime.

## A subtle but important caveat

The configuration model exposes `database.enable_wal_mode`, but the current SQLite store code does not yet apply WAL pragmas automatically. So the flag exists as configuration surface, but it is not fully operational behavior today.
